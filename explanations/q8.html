Explanation

This is a common scenario question on the Google Professional Data Engineer exam, testing knowledge of managed metadata and governance services, specifically Dataplex/Data Catalog, and how they interact with different classes of GCP and external data assets. The key technique here is to match each data source type to the capability of the data catalog service, prioritizing the goal of minimizing development and configuration efforts.

Step 1: Analyze the Goal and Identify the Appropriate GCP Service

The scenario requires improving data discoverability across diverse data assets while minimizing development and configuration efforts. The Google Cloud service designed explicitly for data discovery, centralized metadata management, monitoring, and governance is Dataplex. Dataplex Universal Catalog is the successor to Data Catalog, providing metadata management and search capabilities. Dataplex offers a unified view of all datasets and simplifies data discovery at any scale, making it the foundational tool for this problem.

Step 2: Determine Cataloging Method for BigQuery Datasets

BigQuery is Google Cloudâ€™s serverless data warehouse and a fundamental component of the GCP data ecosystem. Dataplex Universal Catalog automatically creates entries (which represent data assets) for Google Cloud resources, such as BigQuery datasets or tables. This synchronization is continuous and automatic, which aligns perfectly with minimizing configuration effort.

Conclusion for BigQuery: Automatic cataloging.

Step 3: Determine Cataloging Method for Pub/Sub Topics

Pub/Sub is a serverless messaging system provided by Google Cloud. Data Catalog/Dataplex is designed to integrate with core Google Cloud services, explicitly including BigQuery, Cloud Storage, and Pub/Sub, for unified data discovery and metadata capture. Because Pub/Sub topics are considered Google Cloud system resources, their metadata is automatically ingested and kept up-to-date by the Dataplex Universal Catalog service.

Conclusion for Pub/Sub: Automatic cataloging.

Step 4: Determine Cataloging Method for PostgreSQL Tables on Compute Engine

The PostgreSQL instance is running on Compute Engine. While Cloud SQL is a managed relational database service, a PostgreSQL instance running directly on Compute Engine generally represents a custom or external system that is not automatically cataloged as a native system entry by Dataplex. To include metadata from a third-party system, Dataplex Universal Catalog requires custom integration. This involves integrating a custom data source by creating a custom entry using a custom entry type. The metadata must be imported using programmatic methods, such as metadata import API methods via a custom pipeline or by developing a custom connector. This requires manual configuration and development effort to extract metadata from the PostgreSQL instance and push it to the catalog.

Conclusion for PostgreSQL: Manual cataloging using Dataplex APIs or connectors.

Step 5: Evaluate the Options and Select the Best Answer

The required process is:

  • BigQuery: Automatic
  • Pub/Sub: Automatic
  • PostgreSQL: Manual (via API/Custom integration)

Option B accurately identifies the core capability of Dataplex/Data Catalog: it automatically catalogs BigQuery datasets and Pub/Sub topics (core GCP services). It then correctly mandates the use of Data Catalog APIs to manually catalog the metadata for the external asset, the PostgreSQL tables. Since this relies heavily on native, out-of-the-box automation for two of the three assets, it minimizes overall development effort.

Chosen Answer

B. Use Data Catalog to automatically catalog BigQuery datasets and Pub/Sub topics. Use Data Catalog APIs to manually catalog PostgreSQL tables.
