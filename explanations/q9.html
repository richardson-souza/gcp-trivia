Final Answer: C

C. Data will not be lost

The operation of Google Cloud Bigtable relies on a foundational architectural principle where compute is separated from storage. This separation ensures high availability and durability, even during hardware failures.

  • Architectural Design: Bigtable nodes handle client requests and processing, but they do not store the data themselves. Instead, the data is broken into tablets which are stored persistently and durably on Colossus, Google’s distributed file system.
  • Durability: Because the durable storage layer (Colossus) is separate and replicated, if a Bigtable node fails, no data is lost.
  • Recovery: When a node fails, the recovery process is fast because only metadata (the pointers to the tablets) needs to be migrated to a replacement node, rather than copying the actual data.
  • Elimination of D: The service guarantees that if a node fails, the data will not be lost.
  • Elimination of B: The use of the term "Recover data" is inaccurate; the data was never lost in the first place, thus making this option misleading. The surviving nodes or replacement nodes immediately access the persistent data on Colossus.

The Bigtable service is highly available, meaning if a node in a cluster is down, the data is safe because it is maintained by the underlying durable storage system.

This process is like a busy postal sorting office (the node) failing, but all the actual mail (the data) is safely stored in the bomb-proof, redundant warehouse down the road (Colossus). The lost sorting table simply gets replaced quickly, reconnecting to the mail warehouse without losing a single letter.
